# 设置页 CLIProxyAPI 折叠区设计

## 目标

在设置页 `LLM 大模型` 面板中，把 `CLIProxyAPI` 大卡片改成一个默认收起的折叠区。

用户仍然需要使用 `Codex`、`OAuth`、多账号和本地代理统一出口能力，但这些高级控制不应该长期占据 `LLM 大模型` 页面顶部的大块空间。折叠后，主页面先展示常用的厂商 provider 卡片；只有用户需要管理本地代理、OAuth 登录态或上游兼容账号时，才展开 `CLIProxyAPI` 区域。

## 范围

本轮只调整设置页中 `CLIProxyAPI` 区域的展示方式，不改变代理后端能力，也不改变现有接口和账号模型。

包含：

- `CLIProxyAPI` 卡片头部增加折叠/展开交互
- 折叠区默认收起
- 收起时保留标题和简短说明
- 展开时展示现有全部控制项和账号列表
- 前端样式与页面测试同步

不包含：

- `CLIProxyAPI` 后端接口改动
- OAuth 登录流程改动
- Codex 配额逻辑改动
- Agent 账号引用逻辑改动
- 其他 provider 卡片的重构
- 设置页导航或分栏结构调整

## 现状

当前 `renderLlmPanel()` 会直接把 `renderCLIProxyPanel()` 插在 `LLM 大模型` 面板顶部。

这导致：

1. 用户一进入 `LLM 大模型` 页面，就先看到一整块与 `Codex / OAuth / 本地代理` 相关的高级控制
2. 常规 provider 配置区被整体往下推
3. 即便用户不是每次都要操作本地代理，也要持续承受这块信息密度

但在当前项目里，这块能力又不能直接删除，因为：

- `Codex` Agent 会被路由到本地 `CLIProxyAPI` OpenAI-compatible 出口
- OAuth 账号也通过这层统一接入
- 多账号和本地代理出站控制依赖这一层

因此正确方向不是删除，而是收纳。

## 交互设计

### 1. 折叠区结构

`CLIProxyAPI` 保持为 `LLM 大模型` 面板顶部的一张卡片，但卡片主体分成两层：

- 固定可见的卡片头部
- 可折叠的卡片内容区

头部包含：

- `CLIPROXYAPI` eyebrow
- 标题 `Wiki 内置代理与多账号`
- 一句简短说明，明确它用于 `Codex / OAuth / 多账号 / 本地代理统一出口`
- 折叠开关按钮和当前展开状态指示

### 2. 默认状态

页面初次渲染时，`CLIProxyAPI` 折叠区默认收起。

默认收起的原因：

- 用户已明确希望页面默认更清爽
- 常规 provider 卡片是更高频的设置入口
- `CLIProxyAPI` 属于高级能力，应按需展开

### 3. 展开与收起

点击卡片头部的折叠按钮后：

- 如果当前为收起态，则展开内容区
- 如果当前为展开态，则收起内容区

本轮只要求页面级临时状态，不做持久化：

- 不记录到 localStorage
- 不记忆上一次展开状态
- 不跨页面恢复

每次重新进入设置页时，都从默认收起开始。

### 4. 收起态显示内容

收起态下只显示：

- 标题
- 简短说明
- 折叠按钮

以下现有元素全部隐藏：

- `安装 / 更新引擎`
- `启动代理`
- `停止`
- `刷新账号`
- 出站代理 URL 输入框
- OAuth 按钮组
- OpenAI-compatible 上游导入表单
- 账号摘要
- OAuth 账号列表与 Codex 额度区域

### 5. 展开态显示内容

展开态下恢复现有 `renderCLIProxyPanel()` 的全部内容和现有交互：

- 代理安装、启动、停止、刷新
- OAuth 登录按钮
- 授权链接复制按钮
- OpenAI-compatible 上游导入
- 账号摘要
- OAuth 账号列表
- Codex 额度刷新与启停

也就是说，这次只改变“何时显示”，不改变“怎么工作”。

## 结构与实现边界

本轮保持设置页为单文件实现，不拆新模块，不改后端。

预期只修改以下文件：

- `web/client/src/pages/settings/index.ts`
- `web/client/styles.css`
- `test/web-settings-page.test.ts`

实现边界：

- 在 `renderCLIProxyPanel()` 中引入折叠按钮和内容容器
- 为内容容器增加 `hidden` 或等价的显隐控制
- 在设置页绑定逻辑中新增最小的折叠状态切换处理
- 保持现有 `data-cliproxy-*` 控件不变，避免影响已有接口绑定和测试桩

不做：

- `bindCLIProxyControls()` 的接口重构
- `CLIProxyAPI` 数据加载顺序重排
- 现有 `data-cliproxy-*` 选择器改名

## 状态与事件

本轮新增的前端状态只有一项：

- `CLIProxyAPI` 当前是否展开

状态特征：

- 页面级、临时
- 初始值固定为 `false`
- 由折叠按钮点击切换

需要新增或调整的事件只有：

- 折叠按钮的点击事件

现有 `cliproxy` 控件事件全部保留，不改绑定入口。

## 样式设计

样式目标只有两点：

- 收起态足够紧凑，明显弱化大块高级配置
- 展开态延续当前设置页视觉语言，不做重新设计

具体要求：

- 折叠按钮需要有明确的展开/收起方向提示
- 卡片头部在收起态下保持可点击且易识别
- 内容区展开后与现有字段、按钮组、账号列表间距保持一致
- 不引入新的视觉体系，只复用现有 `settings-card` 风格

## 测试设计

需要新增或调整的测试至少覆盖：

1. `CLIProxyAPI` 默认以收起态渲染
2. 收起态下，标题存在，但内容区隐藏
3. 点击折叠按钮后，内容区展开
4. 展开后，现有 `data-cliproxy-install`、`data-cliproxy-oauth="codex"` 等控件仍然存在
5. 展开后，现有 `manages CLIProxyAPI from the LLM settings page` 行为测试仍然通过

测试边界：

- 不验证动画
- 不验证持久化，因为本轮没有持久化
- 不重复覆盖后端接口逻辑，只验证 DOM 与交互入口仍然连通

## 验收标准

- 打开 `LLM 大模型` 页面时，`CLIProxyAPI` 卡片默认收起
- 收起态下，页面顶部只显示 `CLIProxyAPI` 标题、说明和折叠入口
- 点击折叠入口后，现有代理控制、OAuth 按钮、OpenAI-compatible 表单和账号列表全部可见
- 点击后可以再次收起
- 展开后现有 `CLIProxyAPI` 功能仍可使用
- 常规 provider 卡片在默认进入页面时更早出现在视口中
- 设置页现有滚动行为不被破坏
