# 项目日志选区评论入口设计

## 目标

把项目日志页的评论创建入口从顶部固定工具栏改成正文选区旁的浮动按钮。

用户在项目日志正文中选中文字后，应当在选区附近看到一个浮动 `评论` 按钮。点击后，系统继续沿用现有右侧评论栏的编辑流转：创建高亮、展开评论栏、聚焦到新评论输入框，并保留现有的 `保存 / 删除 / 解决` 操作。

## 范围

本轮只调整项目日志页的评论创建交互，不改变评论数据结构，也不新增后端接口。

包含：

- 项目日志正文选区旁的浮动评论按钮
- 选区变化时按钮的显示、定位和隐藏
- 点击浮动按钮后的评论创建流转
- 新建评论后右侧评论栏自动展开并聚焦输入框
- 前端测试和项目日志文档同步

不包含：

- 评论持久化
- 评论线程、回复、提及
- 右侧评论栏整体改版
- 目录栏、筛选逻辑、解决逻辑重构

## 现状

当前项目日志页的评论创建依赖顶部固定工具栏中的 `评论` 按钮。

现有流转是：

1. 用户在正文中选中文字
2. 点击顶部工具栏 `评论`
3. 代码将选区替换为高亮标记
4. 右侧评论栏展开
5. 用户在右侧新评论卡片里输入内容，再点击 `保存 / 删除 / 解决`

问题在于评论动作和正文选区发生位置分离。用户刚选中文字时，视线停留在正文附近，但创建入口在顶部，操作路径不顺。

## 交互设计

### 1. 浮动评论按钮

当且仅当满足以下条件时，显示浮动 `评论` 按钮：

- 当前存在非空选区
- 选区位于项目日志正文 `data-project-log-content` 内
- 选区不是跨出正文区域的无效选择

按钮定位在选区矩形的右上侧附近，优先保证按钮完全落在项目日志页可视区域内。如果靠右或靠上会溢出，则向左或向下收回，但不引入额外定位系统。

### 2. 按钮隐藏规则

以下情况隐藏浮动按钮：

- 选区被清空
- 用户点击到正文外部
- 用户滚动项目日志页
- 用户点击浮动按钮完成评论创建后
- 项目日志内容重新加载

### 3. 点击评论后的流转

点击浮动 `评论` 按钮后，沿用现有评论创建模型：

1. 读取当前选区
2. 创建评论高亮标记
3. 向评论数组头部插入新评论
4. 自动展开右侧评论栏
5. 渲染新评论卡片
6. 将焦点移动到新评论卡片的输入框
7. 更新顶部状态提示文案

这意味着创建入口改了，但保存、删除、解决、筛选仍由现有右侧评论栏承担。

### 4. 顶部工具栏调整

顶部固定工具栏移除 `评论` 按钮，保留：

- `目录`
- `全部评论`
- `未解决`
- `已解决`
- 右侧状态提示

状态提示文案仍然存在，用于表达当前是否已选中文本、是否已创建评论等状态，但它不再承担创建入口。

## 结构与实现边界

本轮维持单文件页面实现，不拆新模块。

预期只修改以下文件：

- `web/client/src/pages/project-log/index.ts`
- `web/client/styles.css`
- `test/web-project-log-page.test.ts`
- `docs/project-log.md`

前端实现边界：

- 在项目日志页根节点内新增一个浮动评论按钮容器
- 通过选区变化和页面事件驱动按钮显隐
- 复用现有 `createCommentFromSelection` 和 `renderComments` 主流程，必要时只做最小参数或返回值补充，以支持创建后聚焦
- 不改评论对象结构，仍使用 `id / quote / text / resolved`

## 状态与事件

页面需要额外维护的临时前端状态只有两类：

- 当前浮动按钮是否可见
- 当前浮动按钮对应的选区锚点位置

建议监听和处理的事件：

- `selectionchange`
- 项目日志页自身的 `scroll`
- 页面根节点内的点击事件

约束：

- 不把整个评论系统改成复杂状态机
- 不新增全局单例
- 不把浮动按钮逻辑扩散到其他页面

## 测试设计

需要把现有项目日志页面测试从“点击顶部评论按钮创建评论”改成“选区旁浮动评论按钮创建评论”。

至少覆盖：

- 选中正文文本后，页面会渲染浮动 `评论` 按钮
- 点击浮动按钮后，会生成高亮标记
- 点击浮动按钮后，右侧评论栏自动展开
- 新评论输入框获得焦点
- 新评论仍可执行 `保存 / 删除 / 解决`
- 样式测试继续确保项目日志页自身为纵向滚动容器

测试不要求验证真实像素定位，只验证 DOM 出现、可点击、主流程正确。

## 验收标准

- 在项目日志正文中选中文字后，选区旁会出现浮动 `评论` 按钮
- 顶部固定工具栏不再提供评论创建按钮
- 点击浮动 `评论` 按钮后，选中文字会被标记成评论高亮
- 右侧评论栏会自动展开
- 新评论输入框会自动聚焦
- 评论仍支持 `保存 / 删除 / 解决`
- 目录和评论筛选按钮保持可用
- 现有项目日志滚动行为不被破坏
