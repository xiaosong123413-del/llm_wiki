# 工具箱设计

## 目标

在工作台新增一个二级导航 `工具箱`，做成真实可管理的本地工具库。工具内容落盘到仓库，支持新增、编辑、删除、搜索、链接跳转，并按固定分类组织：

- 检查清单
- AI工作流
- 模板
- 网站软件

每个条目必须包含一行明确字段：`可解决啥需求`。这行既要在 UI 里展示，也必须进入搜索索引，保证用户可以直接按需求描述搜到条目。

## 信息结构

新增一个顶层目录：

- `工具箱/`
  - `检查清单/`
  - `AI工作流/`
  - `模板/`
  - `网站软件/`

每个条目一份 Markdown 文件，例如：

- `工具箱/检查清单/发布前检查.md`
- `工具箱/AI工作流/文章改写工作流.md`
- `工具箱/模板/周报模板.md`
- `工具箱/网站软件/Figma.md`

## 文件格式

条目继续保存为 `.md`，使用轻量 frontmatter，正文保留自然文档结构。统一格式如下：

```md
---
title: Figma
kind: 网站软件
solves: 快速完成界面设计、原型协作和评审交付
url: https://www.figma.com/
tags:
  - design
  - ui
---

# Figma

## 可解决啥需求

快速完成界面设计、原型协作和评审交付。

## 简介

适合做界面设计、原型和协作评审。
```

约束：

- `solves` 为必填字段
- 正文中同步保留 `## 可解决啥需求` 段落，避免信息只存在 frontmatter
- 搜索同时匹配：
  - `title`
  - `solves`
  - `url`
  - `tags`
  - 正文文本

## 页面结构

`工具箱` 页采用三栏布局：

1. 左侧分类栏
   - 固定四个分类
   - 支持切换分类
   - 支持宽度拖拽

2. 中间列表栏
   - 显示当前分类下的条目
   - 顶部有搜索框
   - 支持新增条目
   - 支持删除条目
   - 每个列表项展示：
     - 标题
     - `可解决啥需求`
     - 标签
     - 是否有外部链接

3. 右侧详情编辑栏
   - 编辑标题
   - 编辑 `可解决啥需求`
   - 编辑链接
   - 编辑标签
   - 编辑正文
   - 保存回 Markdown

## 后端接口

新增一组最小接口：

- `GET /api/toolbox`
  - 返回分类、条目列表、当前摘要字段
- `PUT /api/toolbox`
  - 保存条目内容
- `POST /api/toolbox`
  - 新建条目文件
- `DELETE /api/toolbox`
  - 删除条目文件

后端职责：

- 确保 `工具箱/` 目录和四个分类子目录存在
- 负责 Markdown/frontmatter 读写
- 负责将 `solves` 字段暴露给前端搜索
- 校验路径只能落在 `工具箱/` 下

## 前端行为

- 工作台二级导航新增 `工具箱`
- 默认进入 `项目推进页` 不变
- 打开 `工具箱` 后按分类浏览条目
- 搜索为前端即时过滤，基于接口返回的结构化数据
- 搜索命中规则包含 `可解决啥需求`
- 点击条目后右侧载入编辑态表单
- 保存成功后刷新当前分类列表和详情
- 外部链接提供单独打开按钮

## 搜索规则

搜索关键词统一做小写标准化后，匹配以下字段：

- `title`
- `solves`
- `url`
- `tags`
- `raw`

其中 `solves` 优先展示在列表项摘要位置。这样用户搜“原型评审”“周报复用”“文章改写”时，可以直接命中对应工具。

## 验收标准

- 工作台出现新的二级导航 `工具箱`
- 仓库出现真实的 `工具箱/` 目录结构
- 四类条目可新增、编辑、删除
- 每个条目都有 `可解决啥需求`
- 搜索能命中 `可解决啥需求`
- 保存后内容真实落盘到仓库 Markdown 文件
- 不引入额外存储层，不改成数据库

## 非目标

这轮不做：

- 拖拽排序
- 批量导入
- 富文本块系统
- 多级标签系统
- 云同步
- 权限控制
