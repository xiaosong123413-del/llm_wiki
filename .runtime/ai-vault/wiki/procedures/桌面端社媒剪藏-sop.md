# 桌面端社媒剪藏 SOP

## 1. 问题定义

### 要解决的问题
把用户输入的小红书或抖音链接，稳定采集为本地可保存、可追溯、可继续编译的剪藏源料。

### 适用范围
适用于桌面端已登录目标平台、目标页面可正常访问、允许将正文与媒体保存到本地仓库的场景。

不适用于纯移动端独占内容、已删除内容、无权限内容，以及当前机器缺少下载与写入条件的场景。

### 当前未解决部分
- 平台页面结构突变时，解析逻辑可能失效。
- 抖音资源地址存在时效性，过晚下载可能失效。
- 批量收藏同步时，单条失败仍需要更细的失败归因。

### 成功标准
- 成功生成一篇剪藏 Markdown。
- 成功保存正文主体或至少一个核心媒体。
- 任务状态可追踪。
- 失败时留下明确错误，不允许静默吞掉。

## 2. 如何触发

### 用户触发
- 用户在闪念日记或源料库中输入小红书、抖音分享链接。
- 用户在设置页点击小红书收藏同步。
- 用户在设置页导入 Cookie 后，主动发起单条采集或批量采集。

### 系统触发
- 系统识别输入文本中的首个 URL 为小红书或抖音链接。
- 桌面端注入了可用的 `desktopCapture` 数据。
- 同步流程发现待处理收藏链接并进入采集队列。

### 不应触发
- 未登录目标平台时不触发正式采集。
- 链接不是受支持平台时不进入该 SOP。
- 已有相同任务仍在运行时不重复创建任务。
- 证据不足时不自动升级为标准做法变更。

## 3. 标准做法（已验证 SOP）

### 前置条件
- 桌面端存在可用登录态。
- 设置页已导入或读取对应平台 Cookie。
- 项目可写入 `raw/剪藏` 与 `.llmwiki`。
- 抖音场景下优先保证 `yt-dlp` 可用。

### 操作步骤

#### 小红书
1. 接收分享链接并识别为小红书 URL。
2. 使用桌面登录态请求页面 HTML。
3. 从 HTML 中提取 `window.__INITIAL_STATE__`。
4. 解析标题、正文、作者、发布时间、图片或视频信息。
5. 下载媒体到 `raw/剪藏/assets/<task-id>/`。
6. 生成 `raw/剪藏/<标题>.md`。
7. 写入 `.llmwiki/clip-tasks.json`。

#### 抖音
1. 接收分享链接并识别为抖音 URL。
2. 用桌面隐藏窗口打开页面并等待资源加载。
3. 读取候选视频、音频、标题、作者、描述、日期。
4. 优先使用 `yt-dlp` 下载。
5. 若失败，回退到页面资源直下。
6. 若音视频分离，使用 `ffmpeg` 合流。
7. 生成 `raw/剪藏/<标题>.md` 和本地视频文件。
8. 写入 `.llmwiki/clip-tasks.json`。

### 检查点
- 生成的 Markdown 存在且可打开。
- frontmatter 包含 `platform`、`source_url`、`clip_status`。
- 媒体目录下至少存在一个有效文件，或正文已完整写入。
- 任务 ledger 中存在对应 task 记录。

### 产出物
- 剪藏正文页。
- 本地图片或视频。
- 任务状态记录。
- 可继续进入 source gallery 与 compile 流程的原料。

## 4. 常见问题与待优化方向

### 常见问题
- Cookie 过期。
  表现为页面返回登录拦截或内容缺失。处理方式是重新登录并重新导入 Cookie。

- 页面结构变化。
  表现为能打开页面，但解析不到正文或媒体。处理方式是检查页面状态字段和 DOM 结构是否变更。

- 抖音下载失败。
  表现为 `yt-dlp` 报错或下载后文件为空。处理方式是改走直链下载，必要时使用 `ffmpeg`。

- 标题缺失。
  表现为生成文件名为默认占位名。处理方式是回退到备注首行或任务时间戳，并在后续人工修正。

### 待优化方向
- 批量采集时做更稳定的失败分类。
- 为每个平台增加页面结构变化告警。
- 给任务结果补充更清晰的媒体完整度指标。
- 收藏同步增加去重与断点续跑能力。

## 5. 其他做法与错误原因

### 错误做法 1
- 做法：所有平台统一走通用网页摘要抓取。
- 为什么有人会这么做：实现简单，入口统一。
- 为什么错：社媒内容强依赖登录态、页面状态和资源探测，通用摘要抓不到真实媒体。
- 会导致什么后果：只能拿到残缺文本，视频图文大量丢失。

### 错误做法 2
- 做法：抖音只保存分享文案，不采真实视频。
- 为什么错：这不是剪藏，只是记了一条链接。
- 正确替代方案：保留真实媒体下载链路，把视频或可追溯资源落到本地。

### 错误做法 3
- 做法：采集成功后不记录任务状态。
- 为什么错：失败、部分成功、媒体缺失都无法追踪。
- 会导致什么后果：用户看到表面成功，但仓库里实际是不完整脏数据。

## 7. 版本结论

### 当前推荐版本
- 小红书：`HTML + __INITIAL_STATE__ + 媒体下载`
- 抖音：`隐藏窗口探测 + yt-dlp 优先 + ffmpeg 回退`
- 统一结果：`Markdown + assets + task ledger`

### 废弃内容
- 只存分享文案。
- 只抓封面不抓主体资源。
- 不记录任务状态的伪成功流程。

### 后续观察项
- 平台字段名变更频率。
- 抖音直链回退成功率。
- 收藏同步稳定性。
- `partial` 任务的人工修复成本。
