# 文件同步冲突

文件同步冲突是指在多设备或多客户端环境下，同一文件被不同端同时修改、移动、重命名或删除，导致同步工具无法自动判断以哪个版本为准的状态。在 [[Obsidian生产力系统]] 中，这一问题尤为突出，因为笔记库的文件操作频繁且相互依赖。

## 常见触发场景

以下操作最容易引发同步冲突：

- 在一台设备上**移动文件**到新目录，另一台设备尚未同步时又对该文件进行了编辑
- 在一台设备上**删除文件**，同步工具却在另一端重新创建了同名空文件
- **重命名文件**后，旧文件名与新文件名同时出现在库中，造成内容重复或空文件残留
- 多端同时写入同一文件，产生内容分叉

这类冲突的危险之处在于：用户往往难以区分哪个是有内容的版本、哪个是同步工具自动生成的空文件，极易误删有效内容。

## 坚果云场景下的典型表现

使用坚果云同步 Obsidian 库时，上述问题尤为集中。常见现象包括：

- 移动或删除文件后，坚果云在同步过程中重新生成一个同名的空文件
- 重命名操作导致新旧两个文件同时存在，内容归属不明
- 用户在不确定哪个文件有内容的情况下执行删除，造成数据丢失

## 与 Agent 自动化的关系

在 [[Agent集成工作流]] 中，自动化程度越高，文件同步冲突的风险越需要提前管控。Agent 执行重命名、删除和移动文件等操作时，若同步状态尚未稳定，极易触发上述冲突。因此，必须在 Agent 工具链中预先验证冲突处理机制，而不是在问题发生后补救。

## 缓解建议

- 在执行批量移动、重命名或删除前，确认所有设备的同步状态已完成（坚果云显示"同步完成"）
- 删除文件前，先打开确认其内容不为空
- 对重要文件操作后，检查库中是否出现同名的空文件残留
- 在 Agent 自动化流程中，将文件操作步骤与同步状态检查解耦，避免在同步未完成时触发下一步操作
