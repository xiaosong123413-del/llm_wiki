# LLM应用架构

LLM应用架构是指大型语言模型系统的系统性设计，它将[[Prompt Engineering]]、知识管理、自动化维护和智能体执行层整合为连贯的工作流程。这种架构不是将LLM视为简单的问答工具，而是将其定位为知识操作者，能够在个人和组织知识系统中接管重复性认知任务。

## 核心架构层

### 提示和查询层

基础层专注于高质量的[[Prompt Engineering]]和知识检索。有效的提示依赖于上下文、任务定义、格式规范和约束条件。基于wiki的查询比直接模型交互提供更好的可追溯性，因为它们利用结构化知识而不是从头生成响应。当响应揭示新的结构性见解时，应将其写回知识库以供未来参考。

### 自动化维护层

该层通过自动化流程处理知识系统的持续维护：
- 检测没有传入链接的孤立页面
- 识别损坏的交叉引用
- 标记页面间的矛盾信息
- 建议相关概念之间的相关交叉引用

这些维护功能确保知识库在扩展时保持连贯性和可导航性。

### 智能体执行层

执行层通过本地智能体运行时将LLM从"回答者"转变为"执行者"。这使LLM能够执行文本生成之外的操作，包括文件操作、系统命令和工具集成。智能体层弥合了推理和行动之间的差距，允许LLM参与完整的工作流程而不是孤立的任务。

## [[Compiled Truth]]原则

该架构建立在"[[Compiled Truth]]"概念之上——即LLM应该持续将原始材料编译成可维护的知识页面，而不是执行临时检索。这种方法强调增量编译、交叉链接和持续修订，而不是一次性信息积累。目标是创建随时间改进的知识，而不是简单地存储更多数据。

## 实施考虑

### 按用例选择工具

不同组件在整体架构中服务于不同目的：
- 知识整合需要wiki结构和系统化组织
- 命令执行需要具有工具访问权限的智能体运行时
- 移动访问需要专门的入口点
- 多模型协调需要网关和路由系统

### 与知识管理的集成

LLM应用架构在与更广泛的[[AI知识库构建]]工作流程集成时效果最佳。该架构通过提供在这些层级之间转换信息的计算层，支持三层知识系统（原始材料、来源摘要、正式知识页面）。

## 与规划框架的关系

该架构通过为执行计划任务提供基础设施来补充[[AI Agent Planning Hierarchy]]。虽然规划框架确定要采取什么行动，但LLM应用架构为在不同上下文和工具环境中执行这些行动提供技术基础。
