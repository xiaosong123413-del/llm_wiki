# 文件命名规范体系

文件命名规范体系是指通过统一的命名规则、结构约定和工具辅助，对数字文件进行系统化标识的方法论。合理的命名不仅是文件的"名片"，更是检索、协作、版本追踪和归档的基础。^[inbox（剪藏进来的）__notionmpclipper__2026_关于文件管理的思考_-_少数派__e7626f05.md]

---

## 为什么命名规范至关重要

文件名是检索的前提。以 Everything 为代表的文件搜索工具基于文件名进行匹配，若命名混乱（如 `1`、`111`、`表(1)`），即便文件存在也难以定位。更常见的协作场景包括：需要对比某两个历史版本、追溯某日期的交付物、确认某文件的责任人——这些需求都依赖于结构化的命名信息。^[inbox（剪藏进来的）__notionmpclipper__2026_关于文件管理的思考_-_少数派__e7626f05.md]

---

## 命名设计的核心维度

在设计命名规则时，需要权衡以下五个问题：

- 概括性：能否清晰描述文件内容？
- 可读性：长度是否影响阅读体验？
- 适用性：是否兼容内外部交接场景？
- 检索性：是否便于快速定位？
- 兼容性：是否会引发软件或系统 bug？

^[inbox（剪藏进来的）__notionmpclipper__2026_关于文件管理的思考_-_少数派__e7626f05.md]

---

## 命名结构的七个要素

以下要素可按实际需求组合使用，并非每个文件都需要包含全部字段。推荐以下顺序为基础，根据权重灵活调整：

```
领域_项目_版本_执行人_备注_时间
```

### 1. 领域（Category）

标注文件所属的业务或学科领域。可通过父级文件夹隐含领域信息，避免文件名过长。英文系统或专业软件中建议使用缩写，例如：`DUB`（配音）、`MIX`（混音）、`REC`（录音）、`SD`（声音设计）。^[inbox（剪藏进来的）__notionmpclipper__2026_关于文件管理的思考_-_少数派__e7626f05.md]

### 2. 项目（Project）

文件所属的具体项目名称，是命名的主标题。例如：`DUBG_猿神`（游戏配音项目"猿神"）、`MIXM_新年好`（歌曲混音项目"新年好"）。^[inbox（剪藏进来的）__notionmpclipper__2026_关于文件管理的思考_-_少数派__e7626f05.md]

### 3. 版本（Version）

对需要多次迭代的文件至关重要。使用 `V1`、`V2`、`V3` 递增标注，必要时在括号内补充版本说明：

```
DUBG_猿神_V2(漏配补)
DUBG_猿神_V3(语速慢)
DUBG_猿神_V4(音色亮)
```

^[inbox（剪藏进来的）__notionmpclipper__2026_关于文件管理的思考_-_少数派__e7626f05.md]

### 4. 执行人（Author）

标注文件的创建者或责任人，主要用于团队协作场景。个人工作流中可酌情省略。例如：`_LQ` 代表作者 Lawrence Qin。^[inbox（剪藏进来的）__notionmpclipper__2026_关于文件管理的思考_-_少数派__e7626f05.md]

### 5. 标签（Tag）

在不依赖第三方标签系统的情况下，可通过命名实现类标签功能。优点是极度稳定、可与任意搜索工具配合；缺点是文件名较长且不易复用。例如：`2026_文件管理/知识/信息/体系`。^[inbox（剪藏进来的）__notionmpclipper__2026_关于文件管理的思考_-_少数派__e7626f05.md]

### 6. 备注（Note）

对文件的细节描述，比标签更微观，适合记录版本调整内容、技术参数（如音频的采样率、比特率、声道数）等。^[inbox（剪藏进来的）__notionmpclipper__2026_关于文件管理的思考_-_少数派__e7626f05.md]

### 7. 时间（Date）

推荐使用八位年月日格式 `YYYYMMDD`，可简写为六位 `YYMMDD`。由于系统记录的文件创建时间基于存储时间，可能不准确，在文件名中显式标注时间更为可靠。例如：

```
Glass Sky_SP_V1_CJ_260101      ← 编剧提交初稿
Glass Sky_SP_FB_V1_LQ_260110   ← 导演9天后给出反馈
```

^[inbox（剪藏进来的）__notionmpclipper__2026_关于文件管理的思考_-_少数派__e7626f05.md]

## 命名注意事项

- NTFS 系统单个文件名上限为 255 字符，且从盘符根目录开始计算路径长度，一个汉字占两字节，需注意总长度。
- 各字段顺序可根据实际权重调整，但团队内部必须统一，以保证文件排序的一致性。
- 可在团队内部建立字母缩写映射表，用简短字符传递复杂语义，例如：`OAF`（Original Audio Files，原始音频文件）、`DEAF`（Director Edited Audio Files，导演剪辑音频素材）。
- 尽量避免使用以下特殊字符（影响兼容性）：`! @ # ￥ % & < > ? , { } ' `` 以及空格`。
- 如无必要，尽量避免在文件名中使用中文，以规避部分专业软件的兼容问题。

## 与文件夹结构的配合

命名规范不是孤立存在的，需要与文件夹层级体系协同设计。常见的两种组织方法：

### 杜威十进制法

借鉴图书馆分类逻辑，用数字编号建立层级文件夹体系。顶层分类用两位数前缀（`01`、`02`……）保证排序正确，子分类依次细化。建议文件夹层级不超过六层，每层文件数不超过 99 个。例如：

```
02 工作/
  01 导演/
  02 配音/
    DUBG_猿神_V1_LQ_260101
    DUBG_猿神_V2(漏配补)_LQ_260115
  03 录音/
```

### [[PARA 系统]]

由 Tiago Forte 提出，将文件分为四类：Project（有明确目标和截止日期的项目）、Area（需长期关注的领域）、Resource（支撑项目和领域的知识素材）、Archive（已完成但需留存的归档内容）。Project 文件夹内的文件命名可加时间前缀以便排序，如 `2601_学做咸蛋黄鸡翅`。^[inbox（剪藏进来的）__notionmpclipper__2026_关于文件管理的思考_-_少数派__e7626f05.md]

## 工具辅助

复杂的命名规范在实际操作中可借助工具提升效率：

- **批量重命名工具**（如 ReNamer）：支持批量添加元数据、替换内容、调整大小写、序列化编号、正则表达式操作，可预设常用规则一键应用。
- **专业领域管理软件**：音频文件可用 Basehead 管理几十项参数并批量操作；图像素材可用 Eagle；电子书可用 Calibre。这类工具在软件内部维护元数据，可减少写入文件名的信息量。

## 在知识管理系统中的应用

在 [[Agent集成工作流]] 等自动化场景中，统一的命名规范是 Agent 执行重命名、移动、归档操作的前提。命名混乱会导致冲突检测失效，因此在引入自动化之前，应先在人工流程中验证并稳定命名体系。^[inbox（剪藏进来的）__notionmpclipper__2026_关于文件管理的思考_-_少数派__e7626f05.md]
