# 知识编译 vs 传统 RAG

知识编译（Knowledge Compilation）与传统 RAG（Retrieval-Augmented Generation）是两种根本不同的 LLM 知识利用范式。前者由 Karpathy 提出，核心思想是"编译一次，持续维护"；后者则在每次查询时从原始文档临时检索并拼装答案。两者的差异不仅是技术实现上的，更是对"知识应该以什么形态存在"这一问题的不同回答。

## 核心对比

| 维度 | 传统 RAG | 知识编译（LLM-Wiki） |
|------|---------|------------------|
| 知识组织 | 原始文档 | 编译后的结构化维基 |
| 查询方式 | 每次从头检索 | 在已维护的知识中查询 |
| 更新成本 | 手工维护，簿记负担大 | AI 自动化，增量编译 |
| 质量演进 | 一致但通用 | 递进优化 |
| 可维护性 | 困难 | 容易 |

## 传统 RAG 的工作方式

传统 RAG 系统将原始文档存入向量数据库，每次用户提问时检索相关片段，再由 LLM 基于这些片段生成答案。这意味着每次查询都是"从零拼装"——知识没有被真正消化，只是被临时调取。原始文档的质量、分块方式和检索精度直接决定了答案质量，而这些因素在每次查询中都可能产生不一致的结果。

## 知识编译的工作方式

知识编译模式要求 LLM 维护一个持续演进的结构化维基。新素材进入后，不是直接存档等待检索，而是经过"吸收→编译→连线"的流程，被整合进已有的知识网络。查询时，LLM 访问的是已经经过提炼和结构化的知识页面，而非原始文档片段。有价值的查询分析本身也会被归档为新页面，使探索过程成为知识库的一部分。这一模式对应[[三层知识架构]]中的维基层，是整个系统的核心。

## 为什么编译优于检索

传统 RAG 最大的隐性成本在于簿记工作——更新交叉引用、保持摘要现状、标记矛盾。这些工作繁琐且容易被忽略，导致知识库随时间退化。知识编译模式将这部分工作交给 LLM 完成：LLM 不会厌倦或遗忘，维护成本接近零。人类专注于策展和分析，LLM 负责繁琐的维护，形成清晰的分工。

## 设计原则的差异

传统 RAG 的隐含假设是"知识存在于文档中，检索时取用即可"。知识编译的隐含假设则是"知识需要被消化和编织，才能真正可用"。这一差异体现在具体实践上：知识编译要求 Markdown 结构清晰便于 AI 解析，代码块可直接执行，引用和链接完善以减少额外搜索——这些都是为了让编译后的知识对 LLM 友好，而非仅仅对人类可读。

## 适用场景

知识编译适合需要长期积累、反复查询、知识间存在大量关联的场景——个人知识库、领域专业知识管理、持续学习系统。传统 RAG 在一次性文档问答、知识更新极为频繁或原始文档本身就是最终权威来源的场景下仍有其价值。两者并非完全对立，[[三层知识架构]]中的原始来源层（raw 层）本质上保留了 RAG 的原材料，但长期查询应优先落在已编译的维基层。
