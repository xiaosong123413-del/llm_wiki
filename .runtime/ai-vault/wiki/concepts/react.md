# ReAct

ReAct是一种AI智能体范式，在紧密的反馈循环中交替进行推理和行动。智能体生成一个想法，调用一个工具，观察结果，然后重复这个循环直到任务完成。ReAct解决的是"我下一步应该做什么？"这个问题，而不是提供全面的任务分解，这使得它非常适合需要环境反馈但不需要大量前期规划的交互式任务。

## 核心机制

ReAct通过一个简单但强大的循环运行：
1. **推理**：智能体分析当前情况并决定采取的行动
2. **行动**：智能体执行工具调用或采取行动
3. **观察**：智能体处理结果和环境反馈
4. **重复**：循环继续直到目标达成

这种双向循环使得自我纠正和适应变化条件成为可能，但运行时不需要预先承诺全局计划。

## 在规划层次中的位置

ReAct占据[[Agent Planning Hierarchy]]的中间层，位于[[Chain-of-Thought]]和[[Plan-and-Execute Framework]]之间。虽然CoT专注于改善局部推理质量，[[Plan-and-Execute]]处理全面的任务分解，ReAct专门从事与工具集成的反应式决策制定。

## 适用场景

ReAct特别适用于：
- **信息检索任务**：网络搜索、API调用和数据库查询
- **短期交互工作流**：需要3-5步环境反馈的任务
- **动态环境**：执行过程中条件会发生变化的情况
- **工具密集型操作**：主要涉及调用外部服务的任务

当任务范围有限且不需要复杂的依赖管理或长期战略规划时，这种方法效果最佳。

## 局限性

ReAct的反应性质造成了固有的局限性：
- **无全局优化**：决策是逐步做出的，不考虑完整的任务结构
- **范围有限**：不适合具有依赖关系的复杂多步骤项目
- **潜在低效**：由于缺乏前期规划可能采取次优路径

对于需要全面规划或严格约束执行的任务，[[Plan-and-Execute Framework]]结合[[Harness Engineering]]提供更好的控制和可靠性。

## 工程考虑

实施ReAct时，需要考虑：
- **令牌成本**：相比CoT（低）和[[Plan-and-Execute]]（高），使用量中等
- **工具依赖**：需要强大的外部工具集成
- **错误处理**：反馈循环使从错误中恢复成为可能
- **任务边界**：在明确定义的短期目标下效果最佳

[[Agent Planning Selection Matrix]]为确定何时选择ReAct而非其他规划方法提供了系统性指导。
